Master Plan — Streamlined Build (Clean Slate, MADR Replacement)
0) Product Rule

Statements is the app. Everything happens inside the statement workspace.

Only 3 outputs: Downgrades, Non-PCI, General Audit (derived from extracted fields).

1) User Flow (MVP)

Go to Statements

Select:

Processor family (dropdown)

Client Level (II / III)

Upload statement PDF

Live scan begins → highlights + tables fill in

User can correct extracted values in the left panel (overrides)

Final results instantly recompute

3) Live Scan Pipeline (Deterministic)

Phase A — Classify statement type

Uses processor family rules to locate sections (Summary by Card Type, Fees, Interchange/Program Fees)

Phase B — Extract Identity + Totals

Populate left panel immediately

Phase C — Non-PCI detection

If line contains NON PCI / NON-PCI / NONPCI → outline RED + add to list

Phase D — Downgrade detection

Match interchange lines using processor-specific downgrade library + keyword heuristics

Outline YELLOW + add row to table

Phase E — Compute

Volume, fees, effective rate, revenue lost, rollups

4) Processor-Specific Rulesets (Scalable Architecture)

Everything is keyed by processor_family, e.g.:

fiserv_cardconnect_type1 (your shown format: Fees section only, AMEX carve-out)

Each processor family includes:

Section locators (how to find totals)

AMEX treatment rules

Interchange parsing rules (how to read “TIMES $X”, transactions, amounts)

5) Downgrade Dataset (Core Library)
Table: downgrade_codes

processor_family

card_brand (V/M/…)

raw_label_pattern (string/regex)

category (EIRF, STD, NON_QUAL, PRODUCT_1, CNP, DATA_RATE_1, DATA_RATE_2, LEVEL_II, OTHER)

downgrade_rate (optional)

corrected_rate (optional)

reason_template (optional)

active flag

Runtime behavior

If matched → classified downgrade row (YELLOW)

If keyword-triggered but not in library → still show row Needs Review + enqueue

6) Unknown Handling (Auto → Admin Review)
Table: downgrade_review_queue

statement_id, processor_family, raw_descriptor

suggested_category (from keyword)

extracted volume / fee / page ref

status (new / approved / rejected)

User experience:

Auditor sees the row now (doesn’t block audit)

Admin can later approve and convert it into a new library entry

7) Corrected Rates Strategy (Public standard if available; admin-config otherwise)
Table: rate_rules

processor_family

brand + category or specific downgrade_code_id

corrected_rate

source: public_standard or admin_config

effective date

Runtime fallback:

specific code override

category default

if none → show Revenue Lost as Pending Config (or estimated if you choose)

8) Admin Dashboard (Two screens)
A) Downgrade Library

Filter by processor family / brand / category

Add/edit/disable patterns

Set downgrade rates + corrected rates

B) Review Queue

See “flagged but unknown” descriptors

Actions:

Approve & add to library (choose category + pattern + rate rules)

Reject (not a downgrade)

Merge with existing

9) What replaces MADR (explicitly)

MADR becomes:

downgrade_codes + rate_rules + live statement scanning

Output matches the MADR report style:

Savings/revenue-lost math per row

Status dropdown logic (Congratulations / Pending Investigation / etc.)

Processor+format specific extraction
